# BOX Accounting — Executive Summary

> **Audience:** Anyone joining the AUKI programme who needs to understand what BOX does and how accounting is generated, without reading the full technical reference.  
> **Scope:** Generic BOX accounting process, valid for all products in the AUKI circuit. Product-specific behaviour (e.g., BRS) is documented separately.

---

## What is BOX?

BOX (Back Office Express) is the central back-office platform in the AUKI architecture. It receives trades from Murex, calculates their financial impact, and generates the accounting entries that feed into the bank's official books (the General Ledger). Think of it as a pipeline: a trade goes in at one end, and accounting records come out at the other.

BOX replaces GBO as the accounting source of truth. For each product migrated into the AUKI circuit, BOX ACC takes ownership of generating the OBBs (accounting posting messages) that were previously generated by GBO — or in the case of BRS, by Calypso.

---

## End-to-End Flow: From Murex to MIC

Before reaching BOX, a trade passes through several systems. The full online path from trade booking to accounting generation is:

```
MUREX                   TIBCO EMS               MUREX PBC               CAMUNDA
Trader books    ──XML──>  Message     ──MxML──>  Integration   ──JSON──> Orchestration
trade (MxML)             Bus (EMS)               Layer                   Engine
                                                 (MxML →
                                                 Esperanto)
                                                                             │
                                                                             ↓
                                                                    BOX PBC
                                                                    (Esperanto →
                                                                    BOX format)
                                                                             │
                                                                             ↓
                                                              ┌──────────────────────────┐
                                                              │          BOX             │
                                                              │  Stage 1: Trade Arrival  │
                                                              │  Stage 2: Validation     │
                                                              │  Stage 3: FE Calcs       │
                                                              │  Stage 4: ACC Generation │
                                                              │  Stage 5: MIC Dispatch   │
                                                              └──────────────────────────┘
                                                                             │
                                                                             ↓
                                                                           MIC
                                                                    (Pre-Ledger)
                                                                             │
                                                               ┌─────────────────────┐
                                                               ↓                     ↓
                                                             AC&G               Equation
                                                           (Madrid GL)         (SLB GL)
```

### Step-by-step: the pre-BOX pipeline

**1. Trader books trade in Murex**
The trader executes a deal in Murex (the front-office system and golden source of trade data). Murex is also where trade events (fixings, early terminations, amendments) are executed throughout the life of the trade.

**2. Murex exports an XML message (MxML)**
Murex serialises the trade into its native XML format (MxML) and publishes it to the integration layer.

**3. MxML goes to TIBCO EMS**
TIBCO EMS (Enterprise Message Service) is the message bus that carries the MxML message from Murex to the integration layer. It decouples the front-office system from downstream processing.

**4. Murex PBC integration layer — MxML → Esperanto (JSON)**
The Murex PBC integration layer transforms the raw MxML into Esperanto — the bank-wide canonical message format used across all AUKI integrations. Esperanto is a JSON format standardised by Santander to ensure all systems speak a common language, regardless of product or geography.

**5. Esperanto sent to Camunda via HTTP REST**
The Esperanto JSON message is sent to Camunda via HTTP REST. Camunda is the process orchestrator — the "brain of the post-trade". It receives the Esperanto message and runs BPMN workflows that coordinate the downstream back-office systems (BOX, Integrity Check, Settlements, etc.).

**6. Camunda sends to BOX PBC — Esperanto → BOX format**
Camunda calls the BOX PBC (Packaged Business Capability), which transforms the Esperanto JSON into the BOX internal product format. This layer strips fields that BOX does not need and maps the remaining fields to the BOX data model. The contract for this format is owned by the API Governance team. The result is persisted to the BOX Trade database (`T_BOX_DEAL_LITE_S`), and BOX begins its five-stage accounting process.

## The Five Stages of BOX Accounting

Every trade, regardless of product type, follows the same five-stage path through BOX.

```
Stage 1          Stage 2            Stage 3              Stage 4              Stage 5
─────────        ─────────────      ────────────────      ───────────────      ──────────────────
TRADE            TRADE IS           FINANCIAL             ACCOUNTING           RESULTS SENT
ARRIVES          VALIDATED &        CALCULATIONS          ENTRIES ARE          TO DOWNSTREAM
IN BOX           STORED             ARE RUN               GENERATED            SYSTEMS
```

---

## Stage 1 — Trade Arrival

A trade enters BOX through one of two paths:

**Real-time (Online)**
When a trader executes a deal in Murex, the trade arrives in BOX within seconds via the pipeline described above (Murex → TIBCO EMS → Murex PBC → Camunda → BOX PBC). Inside BOX, the trade processor (`BOX_TRD`) consumes the message from a product-specific Kafka topic. The entry point is a Kafka listener in the trade processor Java service (e.g., `com.santander.box.tradeproc.service.Consumer`). Each product family has its own dedicated trade processor deployment and Kafka topic — there is no single shared processor.

Once received, the trade processor:
- Resolves or creates a BOX deal ID
- Applies optimistic locking to prevent concurrent modifications
- Determines whether the incoming message is a new trade or a material/non-material update to an existing one
- Enriches the trade with static reference data (branch, currency, counterparty details)
- Validates the trade through the Workflow Engine (`BOX_WKF`)
- Persists the trade to the BOX_TRD database via the CRUD service, writing to the Deal Lite table (`T_BOX_DEAL_LITE_S`) and supporting tables (contract, parameters, references)
- Publishes the processed trade to an output Kafka topic for downstream consumption by BOX_FE

**Overnight (Batch — D+1)**
Market data and trade updates arrive in bulk overnight from the Data Lake (Lago). BOX picks them up during its nightly processing run. The Data Lake loads three RAW tables in schema `BOX_FE`:

| Table | Content |
|-------|---------|
| `T_BOX_RAW_DEAL_DATA_S` | Deal structure — contract, dates, legs, currencies |
| `T_BOX_RAW_FLOW_DATA_S` | Cashflows — nominal, interest, premium, cancellations |
| `T_BOX_RAW_MARKET_DATA_S` | Market value and net value |

---

## Stage 2 — Validation and Storage

BOX checks the incoming trade for completeness and correctness:
- Is this a new trade or an update to an existing one?
- Are all required fields present (counterparty, amount, currency, dates)?
- Does the trade match an existing record (duplicate check)?

Once validated, the trade details are stored in BOX's database. Each trade gets a unique identifier and is linked to the correct branch, product type, and counterparty.

For the batch path, `PKG_IMPORT_BOX_FE` loads the RAW table data into the BOX_FE production tables:
- `p_Import_Deal_Data` — MERGE from `T_BOX_RAW_DEAL_DATA_S` → `T_BOX_DEAL_DATA_S`
- `p_Import_Flow_Data` — bulk load from `T_BOX_RAW_FLOW_DATA_S` → `T_BOX_FLOW_DATA_S`

---

## Stage 3 — Financial Calculations

This is the engine room of BOX. For each trade, the Financial Engine (`BOX_FE`, repository `cib-boxfin-dbboxfe`) calculates:

- **Mark-to-Market (MTM):** What is the trade worth today, based on current market rates? This captures unrealised gains or losses.
- **Accruals:** For trades that generate periodic income or expense (like interest on a swap), how much has been earned or owed since the last calculation date?
- **Cash flows:** What payments are due, and when?

The computation is dispatched by `PKG_ENGMAIN_OPTZ.execute_queue_optz` to a product-specific engine package (e.g., `PKG_ENGMAINIR_OPTZ` for IRS, `PKG_ENGMAINMM_OPTZ` for Deposits). Each engine package writes its results to a product-specific financial status table (e.g., `T_BOX_IRFINANCST_S` for IRS). These financial status tables are the input that Stage 4 consumes.

---

## Stage 4 — Accounting Entry Generation

This is where financial calculations become official accounting records. BOX's accounting module (`BOX_ACC`, repository `cib-boxacc-dbboxacc`) runs the following sequence for each trade and event:

**Step 1 — Determine the accounting treatment (Portfolio Properties)**
Based on the trade's product type, branch, and specific conditions, BOX looks up a set of rules called "Portfolio Properties" (`PKG_ACCTPORTFOLIO.f_GetPortfolioProperties()`). These rules define which accounting accounts to use and what type of entries to create. The Topic is then resolved to a GLTA account number (`f_getacctno`).

**Step 2 — Create an Accounting Document**
An Accounting Document is a container that groups all the accounting entries for a single trade on a given date. It gets a unique name and is linked to the original trade. Stored in `T_BOX_ACCT_DOC_S` via `p_createdoc`.

**Step 3 — Post Accounting Movements**
Within each document, BOX creates the individual debit and credit entries. Every financial event (an accrual, a revaluation, a cash payment) results in balanced debit/credit movements posted to specific accounts in the chart of accounts. Stored in `T_BOX_ACCT_MOV_S` via `p_put_mov`.

**Step 4 — Mark the trade as accounted for**
The accounting document is linked back to the trade, so the system knows this trade's financials have been processed for this date (`p_processok`).

The full batch loop is driven by `PKG_BATCHPROCESS_MBJ`, which iterates over Branch / Instrument / Date, runs configured SELECT cursors against the FE output (`p_ExecuteSele`), and calls the product-specific accounting procedures (`p_ExecuteProcs` → e.g., `PKG_IRSACCT` for IRS).

> Key principle: **accounting is by differences — every Debit must equal every Credit (Debit − Credit = 0)**. BOX never posts an unbalanced entry.

---

## Stage 5 — Sending Results to Downstream Systems

Once accounting entries are generated, BOX sends the results to two key downstream systems via the `f_FormatPutMov` function and the Java MBJ layer:

**MIC (the Pre-Ledger)**
MIC receives the accounting entries for inclusion in the bank's official General Ledger. MIC is the gateway to the bank's official books. From MIC, entries are routed to:
- **AC&G** — the General Ledger for the Madrid branch
- **Equation** — the General Ledger for the SLB (London) branch

**Medusa (Liquidity Management)**
Medusa receives cash flow forecasts so the bank's treasury team knows what payments are expected and can manage the bank's cash position.

## A Concrete Example: Interest Rate Swap (IRS)

The following example traces a single IRS trade through the entire BOX process. It is sourced directly from Devin's technical reference and uses illustrative figures.

### What is an IRS?

An Interest Rate Swap is an agreement between two parties to exchange interest payments. For example: "I pay you a fixed rate of 2% per year, and you pay me a floating rate (linked to market rates) every quarter, for 5 years, on a notional amount of EUR 10,000,000."

### Day 0 — The trade is executed

A trader in Madrid agrees to an IRS with a counterparty. Camunda sends the trade details to BOX:
- Notional: EUR 10,000,000
- Fixed rate: 2.00%
- Floating rate: 3-month Euribor
- Start date: today
- Maturity: 5 years
- Payment frequency: quarterly

BOX receives the trade in real-time, validates it, and stores it in the database. The workflow engine assigns it a unique deal ID and routes it through the correct processing steps for an IRS product.

### Day 0 — Financial Engine processes the deal

The Financial Engine immediately runs the initial calculations:
- Calculates the **present value (MTM)** of the swap based on today's interest rate curves
- Sets up the **future cash flow schedule** (20 quarterly payments over 5 years)
- Computes the **first day's accrual** (a fraction of the quarterly interest)

### Day 0 — Accounting entries are generated

BOX_ACC looks up the Portfolio Properties for this IRS and determines which accounting accounts to use and what type of entries are needed.

It creates an Accounting Document (e.g., `IRSBOX12345_65_20240315`) and posts the initial movements:

| Entry | Debit | Credit | Amount |
|-------|-------|--------|--------|
| Record the notional | Notional Account | Counterparty Account | EUR 10,000,000 |
| Record today's MTM | MTM P&L Account | MTM Position Account | EUR 5,230 |

### Every subsequent business day — Overnight batch

Each night, the batch process runs for all live IRS trades:

1. **Accrual calculation:** The fixed leg has accrued EUR 548 of interest (2% / 365 days × EUR 10M). The floating leg has accrued EUR 712 (based on current Euribor). Net accrual: EUR 164 in our favour.

2. **MTM revaluation:** Market rates have moved. The swap is now worth EUR 23,400 (up from EUR 5,230 yesterday). The daily MTM change of EUR 18,170 is recorded.

3. **Accounting entries posted:**

| Entry | Debit | Credit | Amount |
|-------|-------|--------|--------|
| Daily accrual (net) | Accrual Receivable | Interest Income | EUR 164 |
| MTM revaluation | MTM Position Account | MTM P&L Account | EUR 18,170 |

### Quarter-end — Cash settlement

At the end of the first quarter, the actual interest payments are exchanged:
- We pay the counterparty: EUR 50,000 (fixed leg)
- The counterparty pays us: EUR 64,800 (floating leg)
- Net: we receive EUR 14,800

BOX generates settlement entries. The payment instruction is sent to Medusa (for cash management) and the accounting entries go to MIC (for the General Ledger).

### This repeats every day for 5 years

The nightly batch continues to calculate accruals and MTM for this trade every business day until it matures. Each day, new accounting entries are created and the cumulative position is tracked.

## What BOX Needs for a New Product

Before BOX can generate accounting for any new product, three things must exist:

1. **Portfolio Properties configuration** — the rules that tell BOX which accounts to use and what entries to create for each event type. Without this, Stage 4 cannot run.

2. **A product-specific Financial Engine package** — the PL/SQL procedure that calculates MTM, accruals, and cash flows for this product type. Without this, Stage 3 cannot run.

3. **A product-specific Accounting package** — the PL/SQL procedure that translates the FE output into OBBs for this product type. Without this, Stage 4 cannot run.

For BRS specifically, none of these exist today. They are the core deliverables of the BRS project. See `BOX_Accounting.md` for the full technical reference and the BRS section for the complete build list.
